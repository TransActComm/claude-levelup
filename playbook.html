Same line, same gates — whether the change is one file or a feature across services. What scales is how many stages fire and how many hands are on it.

Get it going

Stand it up on your team

The kit is small and mostly convention. None of it is exotic — the discipline is in using it every time, not in the tools.

  1. Install the tooling
    Claude Code, the review bot, and the team plugins — skills for your platform, plus the spec-driven workflow. Connect Slack, GitHub, and your observability provider so a session can read and write the shared surfaces.
  2. Write the doctrine into CLAUDE.md
    The guardrails that stop vibe-coding: evidence before output, no gap-filling, cite the source, production read-only by default. This is what makes the output craft instead of plausible slop.
  3. Make the PR the gate
    Every change is a draft PR with the AI reviewer as first pass. A human graduates it and a second approves — nobody merges their own work. That gate on main is non-negotiable.
  4. Isolate every session in a worktree
    One worktree per branch or task; never branch-switch a shared clone. Isolation is what lets sessions run in parallel without colliding.
  5. Open one coordination channel
    A Slack thread for handoffs, shared state, and announcements; PR comments for the code-anchored record and any privileged asks. Isolated, not siloed.
  6. Design with the same discipline
    Brainstorm → spec → plan as living draft PRs before implementation. Validate the design the way you validate the code.
  7. Close the loop
    Observability (monitors kept in code), CI/CD via your actions runner + chatops, and a habit of feeding every learning back into skills, doctrine, and runbooks. The cycle is the point.
