Step 1 · before a line of design

Find the one force everything bends to

Most requirements here are ordinary. Two are not — and naming them, instead of averaging them away, is what keeps the design honest.

Is the load steady, or a cliff?
"Elastic" undersells it. It's near-idle for weeks, then 10k/sec in an instant. You design for the cliff — admission control — not the average, or the cliff takes you down.
How hard is "never sell a seat twice"?
It's an absolute invariant, not a nice-to-have. That single line forces one owner of truth per seat — every other choice bends around it.
Must "remaining seats" be exact?
No. A browse count that's a few seconds stale is fine — so the huge read load must be kept off the invariant's hot path, served from a cache.
Elastic to what, at what cost?
A fleet sized for 10k/sec sits idle 51 weeks a year. The real ask is burst, not always-on scale — a cost lens, not just a performance one.

Step 2 · what the kata asked for

Two styles, head to head

No best practices, only trade-offs. Each style gets judged against the two forces that matter: the on-sale spike, and the no-oversell invariant.

Space-Based

in-memory data grid, no shared DB on the hot path; scale by adding processing units; persist write-behind

The 10k/sec spike

Its home turf. No central database to bottleneck — the in-memory grid absorbs the burst; add units to scale. This is the textbook case the style was built for.

Never oversell

Seat inventory lives in the grid, partitioned per event so each seat has one in-memory owner. Contention resolves in-memory — fast and correct.

The catch

Operationally heavy: grid replication, collision handling, and a write-behind durability window (a crash before persistence loses sales). Fewer engineers know it cold.

Microservices

catalog, seat-inventory, order/payment, notification as independent services; scale and deploy each on its own

The 10k/sec spike

Each service scales on its own — except the one that can't. The seat-inventory service is a hot partition; a single inventory DB is the bottleneck unless you shard per event and queue in front.

Never oversell

Natural home for the invariant: an inventory service owns seat state with atomic holds / row locks — durable and easy to reason about.

The catch

Hold → pay → confirm spans services, so you inherit a distributed-consistency (saga) problem, and the contended path pays inter-service latency exactly when the spike hits.

The call: neither, whole. Take the seam each wins.

The forces point different ways, so the design does too. The no-oversell hot path takes the Space-Based idea — an in-memory, per-event seat grid with a single owner per seat — because that's what survives 10k/sec without a database bottleneck. Everything calmer (catalog, payments, notifications, post-purchase) stays ordinary microservices, where independent deploy and team familiarity win. Browse "remaining seats" reads from a cache, deliberately off the hot path.

The real unlock isn't the style at all — it's the admission gate in front: a virtual waiting room that sheds and queues load so only what the invariant can safely handle reaches it. Elasticity isn't infinite; you protect correctness by controlling what gets in. Same rule we live by on our own platform — load-shed over retry-storms.

Step 3 · the part that actually ships software

Build → review → prove → ship

Everything above is a spec — it becomes a plan, and both open as living draft PRs before a line of code. For a ticketing on-sale the after-plan half is the whole ballgame: you get one shot at 10k/sec, with no retries. Same concert, traced through the loop.

Build · fan out, one contract

Three sessions, one seam

The seam between the hot sell-path and the calm services was settled in the call (Step 2), so three worktree-isolated sessions run in parallel against that one contract — nobody's waiting on anybody. Handoffs ride a focused Slack thread; the reason behind each contract decision lives in the PR comments, not in someone's head.

Review · the gate, no exceptions

Where the invariant gets defended

Every diff clears the same gate — AI-written code gets more scrutiny, not a pass. On this draw the review catches a real one: the order service read seat availability straight from the browse cache instead of taking a hold through the grid's single owner — a race that oversells the last seat under load. The Step 1 invariant, violated in code, caught at the seam.

Gate: Copilot first → a human graduates it → a second human merges · nobody merges their own work

Prove · the load test is the proof

Done means the fitness function is green

Here's what makes ticketing different: the deciding evidence isn't that it compiles — it's a k6 fitness function. A run at 10,000/sec that asserts zero double-sells and that the admission gate sheds cleanly instead of collapsing. Correctness under the spike is the definition of done; a green build that hasn't survived the load test hasn't proven anything.

Fitness function: 10k/sec · 0 double-sells · gate sheds, never collapses

Ship · rehearsed, then it talks back

One-shot event, backout path first

An on-sale doesn't get a retry, so ship is a rehearsal: the gate dark-launches behind a flag and rollback is one flip — a backout path before the blast radius. Then Datadog watches the real 10k/sec, and the failure a load rig never surfaces — a partner API buckling mid-spike — becomes a doctrine note and a shared skill the next on-sale inherits.
